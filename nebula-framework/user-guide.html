Nebula Agent Framework

Multi-Agent AI Orchestration — End User Guide

What Is This?

Nebula is a multi-agent AI framework that automates software development workflows. Instead of one AI doing everything, it uses specialized AI agents (Product Manager, Architect, Developer, QA, Reviewer) that work in sequence — each passing its output to the next, like a real development team.

Key Concept You give it a feature name (e.g., "user-authentication"), and the framework produces requirements, architecture, code, tests, and a code review — all automatically.

How It Works — The Pipeline

Every workflow follows a strict sequence. Each step must pass its quality gate before the next step begins.

1

Product Manager

Defines WHAT to build — writes user stories, acceptance criteria, non-functional requirements, and out-of-scope items.

🚧 Gate: Stories exist with acceptance criteria
Output → feeds into Step 2
2

Architect

Designs HOW to build it — REST API endpoints, database schema, and architectural decisions.

🚧 Gate: Endpoints + data model defined
Output → feeds into Steps 3, 4, 5
3

Backend Developer

Implements the code — controllers, services, repositories, models, and database migrations following the architecture exactly.

🚧 Gate: At least 1 file with >100 chars + migrations
Output → feeds into Steps 4 & 5
4

QA Engineer

Tests the implementation against acceptance criteria — unit tests, integration tests, and e2e tests.

🚧 Gate: All tests pass (allPassed = true)
Output → final validation
5

Code Reviewer

Reviews code quality — checks for security issues (OWASP), coding standards, and best practices.

🚧 Gate: reviewResult = "approved"
Final step → Workflow complete ✓
What happens if a gate fails? The step retries automatically (up to the configured max retries). If all retries fail, the entire workflow stops and is marked as FAILED.

How to Use the App

Starting the Services

You need to start two services:

ServiceCommandURLPurpose
Backend API npm run dev (in /nebula-framework) http://localhost:4000 Orchestrator engine + REST API
Dashboard npm run dev (in /nebula-framework/dashboard) http://localhost:3000 Visual UI for monitoring & control

Executing a Workflow

  • Open the Dashboard at http://localhost:3000
  • Go to the "Execute Workflow" section
  • Select a workflow type (e.g., feature, plan, build, review)
  • Enter a feature name (e.g., user-authentication)
  • Click Execute — watch the real-time event stream as each agent works
  • View results in the Task Board once complete

Or Use the API Directly

POST http://localhost:4000/api/execute
Content-Type: application/json

{
  "workflow": "feature",
  "featureName": "user-authentication"
}

Quality Gates (Rules)

Each step has a quality gate — a set of validation rules that the AI output must satisfy before moving forward.

GateApplied AfterChecks
requirements-complete Product Manager User stories exist, each has acceptance criteria, non-functional requirements defined
design-approved Architect At least 1 API endpoint, data model defined, architectural decisions documented
code-complete Backend Developer At least 1 file produced, code is substantive (>100 chars), migrations included
tests-pass QA Engineer All test suites pass, test count matches acceptance criteria
review-approved Code Reviewer Review result is "approved", security checks passed
Gate Failure = Retry If a gate fails, the agent retries the LLM call with adjusted context. Default is up to 2-3 retries per step. After all retries are exhausted, the workflow fails.

Role of the LLM (AI Model)

The LLM is the brain behind each agent. Here's what it does at each step:

AgentLLM ReceivesLLM ProducesTemperature
Product Manager Feature name + constraints User stories, requirements, scope 0.7 (creative)
Architect PM's stories + requirements API design, DB schema, decisions 0.4 (balanced)
Developer Architecture + data model Actual TypeScript code files + migrations 0.3 (precise)
QA All previous outputs + criteria Test cases + pass/fail results 0.2 (deterministic)
Reviewer All code files Approved/rejected + findings 0.2 (deterministic)
LLM Provider Chain (Fallback Order)

The system tries providers in order: Azure OpenAI → OpenAI → Mock (simulated). If Azure isn't configured, it falls through to the next available provider.

Important: The LLM always returns structured JSON. The framework never executes any code from the LLM — it only stores and passes the structured output between agents.
