WHAT WE BUILT:

A multi-agent AI workflow system demonstrating how AI agents collaborate through a structured rulebook to automatically generate full-stack applications. This is based on the concepts behind MSIG's "Nebula" framework — where the repeated workflow of Plan → Design → Build → Test is codified once and reused, instead of manually guiding AI every session.

---

THE EXERCISES:

Exercise 1 - Nebula Concepts & Explanation:
We studied the article "The Bet Behind Nebula: When the Prompt Became the Tool" and broke down how the framework solves a real problem — AI forgetting context between sessions and developers repeating the same workflow manually. The key insight: "A prompt becomes a TOOL when it is durable, versioned, reviewable, and connected to real artifacts." We created explanation files and sample code showing how Nebula's 11 agent roles and 9 actions work together.

Exercise 2 - Repo Walkthrough & Implementation:
We walked through the actual nebula-agents repository structure and built a working mini-app that demonstrates the core concepts in action — without needing the full Nebula infrastructure.

---

THE RULEBOOK (What We Created):

The heart of the system is a "Rulebook" — a configuration that defines:

1. ROLES - Each agent has strict boundaries:
   • Product Manager → Defines WHAT to build (cannot write code or pick technology)
   • Architect → Designs HOW to build it (cannot change requirements or write code)
   • Developer → Writes the actual code (cannot change requirements or architecture)
   • QA → Tests everything works (cannot change code or requirements)

2. WORKFLOW - A fixed pipeline:
   Step 1: Product Manager → Step 2: Architect → Step 3: Developer → Step 4: QA

3. GATES - Quality checkpoints between each step:
   • "requirements-complete" → must have user stories before architecture begins
   • "design-approved" → must have API endpoints before coding begins
   • "code-complete" → must have generated files before testing begins
   • "tests-pass" → all tests must pass before feature is shipped

   If any gate FAILS, the pipeline stops. This prevents the #1 problem with AI-generated code: coordination failures where code compiles but doesn't match the spec.

---

THE GENERATED APP:

When the pipeline runs successfully (all 4 gates pass), it auto-generates:
   • tasks-routes.js — A working REST API with GET/POST/DELETE endpoints
   • server.js — Express server ready to run
   • build-summary.json — Full audit trail of what was built and why

---

REACT FRONTEND (Viewer UI):

We also built a React + Vite frontend with 3 views:
   • Rulebook & Workflow tab — Visual pipeline diagram showing agents, gates, and role boundaries
   • Tasks Manager tab — Live CRUD interface hitting the generated Express API
   • Build Summary tab — Shows user stories, endpoints, files generated, and test results

---

WHY THIS MATTERS:

The traditional approach: "Hey AI, help me build a feature" (vague, repeats every session)
The Nebula approach: "Run the plan action for feature X" (structured, reusable, auditable)

This eliminates:
   • Re-explaining context every session
   • AI generating code that looks right but has hidden coordination failures
   • Lack of quality checks until things break in production
